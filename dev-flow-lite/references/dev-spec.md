# 功能开发规范（复杂任务专用）
## 文档说明
- 归属体系：Dev-Flow-Lite
- 关联主文档：SKILL.md
- 适用范围：新增模块、接口开发、数据结构调整、业务逻辑重构、多文件改动、架构优化等复杂任务
- 使用要求：严格按照主文档规则及下方流程执行，本规范需经多方确认后，方可启动开发。

## 一、任务执行流程
整体流程链路：**输出功能规范 → 多方确认 → 编写开发文档 → 分步开发 → 逐项验收 → 迭代归档**

1. **输出功能规范**
完整填写本文档内的标准规范模板，由开发人员联合产品/需求方负责。

2. **等待确认**
出具规范确认回执，由技术负责人联合产品负责人负责。

3. **编写开发文档**
产出接口文档、设计文档、测试用例等，由开发人员负责。

4. **分步开发**
产出阶段性代码并做分段提交，由开发人员负责。

5. **逐项验收**
整理验收清单并标注结果，由测试人员联合产品人员负责。

6. **迭代归档**
整合全套文档、验收报告、版本记录，由开发人员负责。

## 二、标准模板（直接复制使用）
# 标准输出模板
```
【功能开发规范】
1. 迭代目标
2. 业务约束与规则
3. 边界场景与异常处理
4. 详细开发任务拆解
5. 交付验收清单
6. 迭代归档总结
```

## 三、字段逐项填写说明
1. **迭代目标**
关联需求文档/需求单号，清晰描述本次迭代要实现的功能与业务价值，对齐业务预期。

2. **业务约束与规则**
填写业务限制、权限规则、数据规则、性能指标等**可量化约束**。
示例：接口响应耗时 ≤ 300ms、普通用户不可查看后台数据、数据保存有效期 90 天。

3. **边界场景与异常处理**
枚举全部边界/异常场景，并写明对应处理方案，禁止遗漏异常分支。
常见场景：参数为空、参数非法、请求超时、并发请求、数据不存在等。

4. **详细开发任务拆解**
按模块/功能/时间拆分子任务，明确执行顺序，便于分步开发与进度跟踪。

5. **技术依赖与潜在风险**
写明依赖的第三方接口、公共组件、底层服务；同时标注风险点 + 对应应急预案。

6. **交付验收清单**
逐条列出**可落地、可验证**的验收项，作为测试、产品验收的正式依据。

7. **迭代归档总结**
开发完成后填写：计划耗时、实际耗时、遗留问题、后续优化建议。

## 四、配套开发文档强制要求
使用本规范的复杂任务，必须同步产出以下附属文档：
1. 接口文档：新增/改造接口统一录入接口平台，标注入参、出参、错误码；
2. 设计文档：涉及架构、数据表、流程变更时，补充架构/数据库设计说明；
3. 测试用例：梳理正向、反向、边界用例，交付测试人员使用。

## 五、完整填写示例（精简版）
```
【功能开发规范】
迭代目标：REQ-20260612：新增用户积分查询接口，支持前端展示用户当前积分与积分明细。
业务约束与规则：
接口仅登录用户可访问，未登录直接返回 401；
单次查询最多返回 50 条积分明细；
接口响应耗时控制在 300ms 以内。
边界场景与异常处理：
用户 ID 不存在：返回错误码 1001，提示「用户信息不存在」；
分页参数非法：自动修正为默认第 1 页、每页 10 条；
数据库查询超时：重试 1 次，仍失败则返回系统繁忙。
详细开发任务拆解：
定义接口路由与入参出参；
编写数据库查询逻辑；
对接登录权限校验；
联调测试 + 补充接口文档。
技术依赖与潜在风险：
依赖用户基础信息表；风险：大用户积分数据量大可能慢查询，预案：对积分表建立索引。
交付验收清单：
[ ] 登录态正常请求可返回积分数据
[ ] 未登录请求返回 401
[ ] 异常参数、异常用户 ID 按规则返回对应提示
[ ] 接口响应耗时符合性能要求
迭代归档总结：（开发完成后填写）
```

## 六、使用约束
1. 本文档仅用于 `SKILL.md` 定义的**复杂任务**，简单改动禁止使用本模板；
2. 所有字段必须完整填写，边界场景、风险项不得省略；
3. 规范未完成多方确认，禁止启动编码工作；
4. 开发、验收、归档全流程必须与本规范保持一致，需求变更需重新走确认流程；
5. 全套规范+附属文档（接口/设计/用例）统一归档至项目文档库。

# Dev-Flow-Lite 
轻量化开发工作流，适用代码新增、修改、Bug 修复、样式调整、逻辑重构、功能迭代等**研发代码类任务**，

## 一、核心规则
1. **增量开发**
   仅完成当前需求/问题范围内改动，**禁止私自新增功能、全局优化、修改无关文件/逻辑**；无注释、无记录的隐性改动视为违规。
2. **分级执行**
   按任务复杂度区分流程：简单任务使用 `dev-plan.md`，复杂任务使用 `dev-spec.md`，禁止混用流程。
3. **闭环流程**
   所有任务必须走完「规划确认 → 落地开发 → 自查校验 → 归档记录」完整链路，严禁跳步、缺环节。
4. **可验可溯**
   每一次代码改动必须留存**改动依据、验收标准、版本记录**，支持事后追溯与问题复盘。

---

## 二、任务分级标准（量化判定）
满足任意一条即可判定为对应等级，并使用配套文档。

### 简单任务（轻量提案模式）
判定条件：
- 改动文件数量不超过 1 个
- 代码改动行数不超过 50 行
- 不存在跨模块、接口、数据结构变更

典型场景：Bug 修复、样式调整、参数修改、单文件逻辑纠错、注释或文案修正。
配套文档：`dev-flow-lite/references/dev-plan.md`

### 复杂任务（完整规范模式）
判定条件：
- 改动文件数量不少于 2 个
- 代码改动行数不少于 100 行
- 涉及新增模块、接口、数据表
- 包含业务逻辑重构、架构调整、跨模块联调

典型场景：新功能开发、接口改造、数据结构变更、多文件迭代、架构优化。
配套文档：`dev-flow-lite/references/dev-spec.md`

---

## 三、强制铁律
1. 禁止无规划直接写代码
2. 禁止超需求开发、私自新增功能
3. 禁止隐蔽、无记录的代码改动
4. 禁止忽略边界场景与异常处理
5. 禁止省略自查、归档闭环步骤
